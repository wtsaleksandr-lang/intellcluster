Authentication

The public API currently operates under the same session model as the web UI. Sign in at /login — the magic link drops a intellcluster_user cookie that your client can then attach on subsequent requests. For server-to-server integrations, bearer-token auth is on the roadmap; contact us for early access.

Anonymous requests are supported for experimentation but rate-limited tighter and cannot access history. Signed-in users see run history on /account.

Rate limits

Sliding 60-second window, per bucket. Bucket key is user email when signed in, IP address otherwise. Every response carries the limit state in headers:

X-RateLimit-Tier:      pro
X-RateLimit-Limit:     300
X-RateLimit-Remaining: 287
X-RateLimit-Reset:     1745424180
TierRequests / minute
anon / free30
starter120
pro300
team600

On 429, we return Retry-After in seconds. Back off linearly; never tight-loop.

Phronesis — POST /phronesis/api/decide

POST /phronesis/api/decide

Rank a set of options against weighted criteria using three blind analyst models. Returns ranked options, confidence, agreement, and per-dimension scores. Supports SSE streaming via Accept: text/event-stream; otherwise returns JSON when done.

Request body

{
  "question":   "Which CRM should a 15-seat SaaS team pick?",
  "options":    ["HubSpot", "Salesforce", "Pipedrive"],
  "criteria": [
    { "name": "Lead capture",    "weight": 20 },
    { "name": "Reporting depth", "weight": 15 },
    { "name": "Total cost (12m)","weight": 15 },
    { "name": "Ease of admin",   "weight": 10 }
  ],
  "settings": {
    "depth":      "standard",
    "focus":      "balanced",
    "length":     "standard",
    "web_search": false
  },
  "attachments": []
}

Field constraints: question 5–500 chars. options 2–10. criteria[].weight 1–10. settings.depth is one of quick/standard/deep; focus is balanced/risks/practical; length is concise/standard/detailed.

Response body (JSON mode)

{
  "run_id":           "a1b2c3d4",
  "question":         "...",
  "winner":           "HubSpot",
  "why_winner_won":   "...",
  "judges_agree":     true,
  "judge_count":      3,
  "confidence_level": "moderate",
  "confidence_score": 72.0,
  "total_cost_usd":   0.0423,
  "latency_ms":       14890,
  "ranked_options": [
    {
      "option":            "HubSpot",
      "rank":              1,
      "final_score":       7.84,
      "dimension_scores":  { "Lead capture": 8.3, "Reporting depth": 7.1, ... },
      "strengths":         [ "...", "..." ],
      "weaknesses":        [ "...", "..." ],
      "rank_points":       6
    },
    ...
  ]
}

Use the returned run_id to link to a shareable result at /phronesis/result/{run_id}, download a PDF at /phronesis/result/{run_id}.pdf, or fetch the per-decision OG card at /og/phronesis/{run_id}.png.

Phronesis — POST /phronesis/api/extract

POST /phronesis/api/extract

Natural-language → structured input extractor. Pass free-form text, get back {question, options, criteria} suitable for posting to /decide.

Request:  { "text": "I can't decide between HubSpot and Salesforce for our 15-person sales team..." }
Response: { "question": "...", "options": [...], "criteria": [...] }

Returns 422 if the text couldn't be parsed into a valid decision shape.

Synthesis — POST /synthesis/api/run

POST /synthesis/api/run

Five-model orchestrated research brief. Always returns SSE — the response streams progress events, per-model status, strategist extractions, and the final merged brief. See SSE format below.

Request body

{
  "prompt":     "Compare RAG vs domain-specific fine-tuning for a 10M-doc startup.",
  "category":   "architecture",
  "mode":       "standard",
  "quick_mode": false
}

prompt 10–8000 chars. category drives orchestration shape — valid values: deep_research, architecture, strategy, competitive, risk, and a handful more listed in the FAQ. mode is standard (five mid-tier models) or expert (five flagship models, roughly 3× the cost). quick_mode runs a single research phase instead of the full pipeline.

Response — SSE events

  • phases_info — the phase names the strategist will run through
  • model_start / model_end — per-model lifecycle with latency
  • strategist — intermediate outputs per phase
  • decision — the final merged brief (includes run_id)
  • done — pipeline complete
  • error — terminal, stream ends

Final brief URL: /synthesis/result/{run_id}. Use it to share or re-render.

Templates — GET /api/templates

GET /api/templates

List every decision template we've seeded. Each row has a slug you can fetch individually at /api/template/{slug}. The landing page is /templates.

GET /api/templates
→ { "templates": [ { "slug": "choose-laptop", "title": "...", "options": [...], "criteria": [...] }, ... ] }

Compare presets — GET /api/compare-preset/{slug}

GET /api/compare-preset/{slug}

Returns a pre-configured Phronesis input for a known comparison (e.g. hubspot-vs-salesforce-vs-pipedrive). Feed the returned payload straight into /decide.

Health — GET /api/health

GET /api/health

Liveness + provider status. Bypasses the rate limiter so you can poll it for uptime monitoring.

{
  "status":        "ok",
  "tools":         ["phronesis", "synthesis"],
  "providers":     ["openai", "anthropic", "google", "deepseek", "xai"],
  "has_api_keys":  true
}

Error model

Every error response is JSON with error set to a short string. For rate limits, we also include retry_after_seconds and tier.

StatusWhen
400Malformed request body
413File upload exceeds 5 MB
422Input couldn't be parsed (e.g. extractor)
429Rate limit exceeded. Inspect Retry-After.
500Upstream model failure. Branded with a ref ID.
503No provider API keys configured server-side

SSE streaming format

All streaming endpoints emit standard Server-Sent Events with named event types. Each frame is event: <name>\ndata: <json>\n\n. Your client should listen by event name, not by order. A typical Synthesis run emits ~30–80 frames.

event: phases_info
data: {"names":["Frame extraction","Reconciliation","Brief"]}

event: model_start
data: {"model":"gpt-4o-mini","phase":1}

event: strategist
data: {"phase":1,"result":"..."}

event: decision
data: {"run_id":"a1b2c3d4","result":"..."}

event: done
data: {}

In Python, httpx-sse or the built-in aiter_sse() from httpx both work. In JS, EventSource is simplest but doesn't let you set Accept — use fetch with a manual reader instead (our web UI does this in /phronesis).
